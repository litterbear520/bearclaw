对话压缩
三层防线

LLM 的上下文窗口是有限的。当对话历史超过窗口容量，agent 就会"失忆"。nanobot 设计了一套三层渐进式压缩架构——从单次调用内的实时裁剪，到跨轮次的 LLM 摘要，再到空闲时的后台归档——确保对话永不丢失关键上下文。

核心模块 3 层
源文件 4 个
关键常量 12+
为什么需要压缩？

LLM 的上下文窗口既是记忆，也是瓶颈。nanobot 作为持久化多轮对话 agent，必须处理三个核心挑战：

⛔

窗口溢出

模型有固定的 context window（如 200K tokens）。一个长对话加上工具调用结果，很容易超出限制。超出后，API 直接报错 —— agent 彻底瘫痪。

💸

成本失控

即使没超限，每轮对话都把完整历史发给 LLM，token 用量线性增长。consolidation_ratio = 0.5 的设计目标是把 prompt 控制在窗口的 50% 以内。

🧠

注意力稀释

即使塞得下，过长的 prompt 会让模型"注意力稀释"——关键指令被淹没在大量旧工具输出里。精简上下文反而让模型表现更好。

三层渐进式架构

nanobot 不是一次性砍掉旧消息，而是设计了三道逐步加深的防线。每一层有不同的触发时机、压缩策略和作用范围，从"无损裁剪"到"有损摘要"逐级递进。

L1
实时

In-flight Microcompact

context_governance.py → ContextGovernor

单次 LLM 调用内的实时裁剪。当 runner 即将发送请求时，ContextGovernor 检查 prompt 是否超出窗口，按策略压缩旧的工具返回结果。只影响发给模型的副本，不修改持久化历史。

这是最快、最安全的防线——零延迟、零 LLM 调用、完全可逆。

L2
轮次间

Token-budget Consolidation

memory.py → Consolidator

轮次之间的主动压缩。在每轮对话的前后，Consolidator 估算当前 session 的 prompt 大小。如果超过安全预算（context_window × consolidation_ratio），就把旧消息块通过 LLM 总结成摘要，写入 history.jsonl，并推进 last_consolidated 指针。

这是核心的"有损压缩"层——用一次廉价 LLM 调用换取大量 token 节省。

L3
空闲

AutoCompact Idle Archival

autocompact.py → AutoCompact

会话空闲时的后台归档。当一个 session 超过 session_ttl_minutes（默认 15 分钟）没有新消息，AutoCompact 自动触发：把大部分历史归档为摘要，只保留最近 8 条消息。下次用户回来时，摘要作为 [Archived Context Summary] 注入到 system prompt 中。

这是最激进的压缩——为长期闲置的 session 释放存储和 token 开销。

Layer 1

In-flight Microcompact

在单次 LLM 调用内实时裁剪上下文。不修改持久化历史，只影响发给模型的副本。

工作流程
  1. 清理异常消息
    strip_placeholder_assistant_messages() 移除压缩占位符；strip_malformed_tool_calls() 清除无效工具调用。
  2. 修复消息依赖
    drop_orphan_tool_results() 删除无主工具结果；backfill_missing_tool_results() 为缺失结果补占位符。
  3. 裁剪工具输出
    apply_tool_result_budget() 对超长工具结果截断或持久化到磁盘。然后 compact_inflight_overflow() 把旧的 COMPACTABLE_TOOLS 结果替换为摘要句。
  4. 截断历史
    snip_history() 从历史头部按 token 预算截断，保证以 user 消息开头且不破坏 tool_call 边界。

⚡ 触发条件

每次 AgentRunner 即将调用 LLM 时自动执行
estimated_tokens > input_budget
input_budget = context_window - max_tokens - 1024

Layer 2

Token-budget Consolidation

在轮次之间使用 LLM 摘要旧消息，推进 last_consolidated 指针，实现"有损但高质量"的压缩。

工作流程
  1. 估算 token 用量
    estimate_session_prompt_tokens() 构建完整 probe 请求（包含 system prompt、历史、工具定义），通过 provider 计数器或 tiktoken 估算总 token 数。
  2. 检查是否超预算
    如果 estimated < budget（context_window - max_tokens - 1024），无需压缩。否则设定 target = budget × consolidation_ratio（默认 0.5），开始压缩循环。
  3. 选择压缩边界
    pick_consolidation_boundary() 从 last_consolidated 开始遍历，在 user 消息边界处切割，确保切割的 token 量 ≥ tokens_to_remove。
  4. LLM 摘要归档
    archive() 将被切消息格式化后发给 LLM，用 consolidator_archive.md 模板提取关键事实（SNIP 标准），摘要写入 history.jsonl。失败则 raw_archive() 原文转储。
  5. 推进指针 & 循环
    session.last_consolidated = end_idx，保存 session。如果 estimated 仍超 target，继续下一轮（最多 5 轮）。最终将 _last_summary 存入 metadata。

摘要模板：SNIP 标准

LLM 用 consolidator_archive.md 模板对被归档的消息进行评估。只有满足 SNIP 四项标准的事实才值得保留：

标准 含义 示例
Signal 如果遗忘，用户需要重复说明吗？ 用户偏好、纠正过的错误
Novel 不是同一段对话中其他事实的重述 新发现的技术约束
Important 能防止返工或记录偏好/规则 项目架构决策
Persistent 2 周后仍然相关 配置参数、API 密钥位置

每条事实标记为 [permanent]、[durable]、[ephemeral]、[correction] 或 [skip]。 最终由 Dream 阶段进一步整合到 SOUL.md / USER.md / MEMORY.md 等持久文件中。

Layer 3

AutoCompact 空闲归档

当 session 空闲超过阈值，自动在后台压缩全部历史，只保留最近尾部。

工作流程
  1. 检测空闲 session
    AgentLoop 的消息总线每次 1 秒超时时，调用 auto_compact.check_expired()。遍历所有 session，检查 updated_at 是否超过 session_ttl_minutes（默认 15 分钟）。跳过活跃 session 和内部 session（dream: 前缀）。
  2. 检查可压缩内容
    _has_compactable_idle_tail() 构建一个 probe Session，对未压缩尾部调用 retain_recent_legal_suffix(8)，检查是否有消息可以被移除。
  3. 后台异步归档
    compact_idle_session() 在 consolidation lock 下执行：分离出要保留的 8 条消息和要归档的旧消息。调用 archive() 让 LLM 对全部消息（包括保留的）生成摘要，但只 raw-dump 被移除的。
  4. 注入摘要上下文
    摘要存入 session.metadata["_last_summary"]。下次用户回来时，prepare_session() 读取摘要，ContextBuilder 把它作为 "[Archived Context Summary]" 注入 system prompt。

class AutoCompact:
    _RECENT_SUFFIX_MESSAGES = 8

    def check_expired(self, schedule_bg,
                      resolve_runtime, active_keys):
        # 遍历所有 session
        for info in self.sessions.list_sessions():
            key = info["key"]
            if key in active_keys:
                continue
            if self._is_expired(info["updated_at"]):
                if self._has_compactable_idle_tail(key):
                    schedule_bg(self._archive(key))

    def prepare_session(self, session, key):
        # 用户回来时注入摘要
        entry = self._summaries.pop(key, None)
        if entry:
            return session, self._format_summary(
                entry[0], entry[1])

        # 冷启动：从 metadata 恢复
        meta = session.metadata.get("_last_summary")
        if isinstance(meta, dict):
            return session, self._format_summary(
                meta["text"], ...)
        return session, None

关键数据结构

理解压缩机制的核心在于理解 Session 对象上的几个关键字段，以及它们如何协同工作。

字段 / 常量 位置 说明
session.messages Session 完整的消息列表。新消息 append 到末尾，旧消息通过 last_consolidated 指针标记为"已归档"但仍保留在列表中（L2），或被 AutoCompact 直接移除（L3）。
session.last_consolidated Session 整数指针。指向 messages 列表中第一条"未归档"消息的索引。L2 压缩时向前推进此指针。get_history() 只返回指针之后的消息。
metadata._last_summary Session.metadata 字典 {"text": "摘要内容", "last_active": "ISO时间"}。由 L2/L3 写入。prepare_session() 读取后注入 system prompt。进程重启后仍可从 JSONL 文件恢复。
history.jsonl workspace/memory/ 追加写入的 JSONL 文件。每条记录 {cursor, timestamp, content, session_key}。L2 的 archive() 写入 LLM 摘要，失败时 raw_archive() 写入原文转储。Dream 周期性消化此文件。
compacted_tool_call_ids AgentRunner set[str]。一次 agent 执行期间追踪哪些 tool_call_id 已被 L1 压缩。避免重复压缩。ContextGovernor 读写此集合。
consolidation_ratio Config 浮点数，默认 0.5。L2 的 target = budget × ratio。值越小，压缩越激进（保留更少历史）。范围 [0.1, 0.95]。
FILE_MAX_MESSAGES session/manager.py = 2000。session 文件的硬上限。enforce_file_cap() 在超出时触发，归档超出部分后截断。这是物理存储的保护机制。
replay_max_messages Consolidator 基于 context_window_tokens 计算的自适应回放窗口。= min(2000, max(120, context_window / 100))。超出的消息通过 _consolidate_replay_overflow() 归档。

设计精髓

三层压缩的本质是空间换时间，摘要换原文。L1 不丢信息（只裁剪副本），L2 有损但高质量（LLM 摘要），L3 最激进但只针对冷数据。每一层都有降级策略——LLM 失败时 raw_archive()，确保消息永不丢失。
