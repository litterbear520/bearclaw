Roadmap

从零到一百，构建一个 Agent 项目

以 nanobot 源码为蓝本，模拟作者视角，用第一性原理一步步理解：Agent 是什么，为什么这样设计，如何从一个 50 行脚本长成一个完整框架。

Phase 0

第一性原理：Agent 到底是什么

在写任何代码之前，先搞清楚一个根本问题。

传统程序的逻辑是你写死的——if/else、switch/case，每一步做什么由程序员决定。用户输入 A，走分支 A；输入 B，走分支 B。程序永远不会"想"出你没写过的分支。

第一性原理
Agent = 一个程序，它用 LLM 来决定自己的下一步动作，而不是靠硬编码的逻辑。

这一个定义就够了。从这句话出发，所有的设计决策都会自然浮现：

三个最小组成部分

1. LLM（大脑）——理解输入，决定下一步做什么。它不执行，只决策。

2. Tools（手脚）——LLM 能调用的能力：读文件、跑命令、搜网页。没有 Tools 的 LLM 只能说话，不能做事。

3. Loop（循环）——把 LLM 和 Tools 串起来的控制结构。发消息 → LLM 思考 → 要么给出最终回复，要么请求调用某个工具 → 执行工具 → 把结果反馈给 LLM → 再次思考 → 循环，直到 LLM 认为任务完成。

┌───────────────────────────────────────────────┐ │ Agent Loop │ │ │ │ 用户消息 ──→ LLM ──→ 最终回复？──→ 返回 │ │ │ ↑ │ │ │ tool_call │ 工具结果 │ │ ↓ │ │ │ Tool ──────────┘ │ │ │ └───────────────────────────────────────────────┘
为什么这个区分重要？

一个 chatbot（聊天机器人）只有 LLM，没有 Tools 和 Loop——它只能回答问题，不能采取行动。Agent 的本质是能行动，而且自主决定采取什么行动。这就是为什么 nanobot 的核心只有两个文件：loop.py（循环）和 runner.py（执行引擎），而不是一个巨大的路由表。

一个思想实验

假设你让一个 Agent "帮我把这个 Python 项目的所有 print 改成 logging"。Agent 不需要你告诉它每个文件在哪——它会：

  1. 调用文件搜索工具，找到所有 .py 文件
  2. 逐个读取文件内容
  3. 判断哪些文件有 print 语句
  4. 调用文件编辑工具替换
  5. 跑一下测试确认没改坏

每一步做什么，是 LLM 在运行时动态决定的，不是程序员预先写好的。这就是 Agent 和传统 CLI 工具的根本区别。

nanobot 为什么叫 "nano"？

因为核心理念就是最小化——Agent Loop 本身应该尽可能小、尽可能简单。复杂性不在循环本身，而在循环外面挂载的能力（Tools）和连接的外部世界（Channels）。框架的职责是做好这个循环和插件机制，不应该替 LLM 做决策。

Phase 1

最小 Agent 循环

50 行代码，一个能调用工具的 Agent 就跑起来了。这是一切的起点。

不要从框架开始。从一个能跑的最小脚本开始。

minimal_agent.py — 整个 Agent 的原型
import openai

client = openai.OpenAI()

# 定义一个工具：读取文件
tools = [{
    "type": "function",
    "function": {
        "name": "read_file",
        "parameters": {"type": "object", "properties": {
            "path": {"type": "string"}
        }}
    }
}]

def execute_tool(name, args):
    if name == "read_file":
        return open(args["path"]).read()

messages = [{"role": "user", "content": "读一下 main.py 的内容"}]

# 这就是 Agent Loop —— 整个框架的心脏
while True:
    response = client.chat.completions.create(
        model="gpt-4o", messages=messages, tools=tools
    )
    msg = response.choices[0].message

    # 没有 tool_call → LLM 认为任务完成，返回最终回复
    if not msg.tool_calls:
        print(msg.content)
        break

    # 有 tool_call → 执行工具，把结果喂回去
    messages.append(msg)
    for call in msg.tool_calls:
        result = execute_tool(call.function.name, call.function.arguments)
        messages.append({
            "role": "tool",
            "tool_call_id": call.id,
            "content": result
        })
核心认知
这 50 行代码就是一个完整的 Agent。nanobot 的 runner.py 有 600+ 行，但核心逻辑和这个一模一样——多出来的全是错误处理、流式输出、重试、上下文治理。骨架没变。

对应 nanobot 源码

这个最小循环在 nanobot 里被拆成两层：

  • agent/runner.py — AgentRunner 类，纯粹的 "LLM 对话 + 工具执行" 循环。不关心消息从哪来、回复往哪发。
  • agent/loop.py — AgentLoop 类，在 Runner 外面包了一层：管理会话、构建上下文、处理 Hook。

为什么拆成 Loop + Runner 两层？

因为关注点分离。Runner 只管 "和 LLM 对话 + 执行工具"，它不知道消息来自 Telegram 还是 WebUI，也不管会话历史怎么存。Loop 负责这些"产品层"的事情。这样 Runner 可以被单独复用——比如子 Agent（subagent）直接调用 Runner，不需要再走一遍 Loop 的全部流程。

这个阶段你学到什么

  1. while True 循环是 Agent 的心脏
  2. LLM 的 tool_call 机制是 Agent 能"行动"的根本
  3. 循环的退出条件：LLM 返回纯文本（没有 tool_call）= 它认为任务完成了
  4. 所有后续的架构都是在这个骨架上"长"出来的
Phase 2

工具系统

从 if/else 硬编码到可插拔的注册表——Agent 的能力如何扩展。

Phase 1 的 execute_tool 用 if/else 硬编码了工具。加一个工具就要改这个函数。三个工具还行，三十个就崩了。

设计原则
工具应该是可插拔的。加一个工具 = 写一个文件放进目录，框架自动发现并注册。不需要改任何已有代码。

三层抽象

Tool 基类 agent/tools/base.py ——每个工具继承它，声明自己的名字、描述、参数 JSON Schema，实现一个 execute 方法。

ToolRegistry agent/tools/registry.py ——一个字典，key 是工具名，value 是 Tool 实例。Runner 调用 registry.execute(name, args)，不关心具体是哪个工具。

Auto-discovery agent/tools/loader.py ——启动时用 pkgutil 扫描 tools/ 目录下的所有模块，自动实例化并注册。

一个典型的 Tool 长什么样
class ReadFileTool(Tool):
    name = "read_file"
    description = "读取文件内容"

    def parameters(self):
        return {
            "path": Param(type="string", required=True)
        }

    async def execute(self, path: str) -> str:
        return Path(path).read_text()

为什么用 JSON Schema 描述参数？

因为 LLM 的 tool_call 协议就是 JSON Schema。LLM 需要知道：这个工具叫什么、接受什么参数、每个参数是什么类型。你用 JSON Schema 描述，框架自动转换成 LLM 能理解的工具定义，同时也能自动校验 LLM 返回的参数是否合法。一份 Schema，两个用途。

nanobot 内置了哪些工具

都在 agent/tools/ 目录下，每个文件一类工具：

  • filesystem.py — 文件读写编辑
  • shell.py — Shell 命令执行（带沙盒）
  • web.py — 搜索和网页抓取
  • search.py — 代码搜索（grep/glob）
  • mcp.py — MCP Server 协议
  • spawn.py — 子 Agent 生成
  • cron.py — 定时任务
  • long_task.py — 持续目标追踪
插件机制的威力

想给 Agent 加一个"发邮件"的能力？写一个 email.py，继承 Tool，放进 tools/ 目录。重启后 Agent 就自动拥有了发邮件的能力——LLM 会在需要时自己调用它。你不需要在任何地方写 "如果用户说发邮件就调用 send_email"。

Phase 3

Provider 抽象

不要绑死一个 LLM——今天用 GPT-4，明天可能用 Claude，后天可能用本地模型。

Phase 1 里硬编码了 openai.ChatCompletion。但不同 LLM 的 API 格式不同、认证方式不同、流式输出方式不同。

设计原则
Runner 不应该知道自己在跟哪个 LLM 说话。它只调用 provider.send_message(messages)，返回一个统一的 LLMResponse。

三件事

LLMProvider 基类 providers/base.py ——定义接口：send_message(messages) → LLMResponse。LLMResponse 包含 content（文本回复）和 tool_calls（工具调用请求）。

具体 Provider ——每个 LLM 一个实现：Anthropic、OpenAI Compatible、Azure、Bedrock 等。各自处理自己的 API 差异。

Factory + Registry providers/factory.py ——根据模型名（如 claude-sonnet-4-20250514）自动找到对应的 Provider 并实例化。

nanobot 的选择：统一接口 + 自动路由

用户在 config 里写模型名，框架自动选 Provider。切换模型 = 改一行配置。

nanobot 采用的方案
另一种方案：LiteLLM 代理

用第三方库统一所有 LLM API。优点是省事，缺点是多一层依赖、调试困难、流式处理受限。

为什么 nanobot 自己实现而不用 LiteLLM？

因为 Agent 框架对 LLM 交互有更精细的需求——流式输出、tool_call 解析、重试策略、thinking/reasoning 提取、provider-specific 字段传递。这些在一个通用代理层里很难控制。自己实现虽然代码多一些，但完全掌控每个细节。

Phase 4

记忆与会话

没有记忆的 Agent 像金鱼——每次对话都是全新的。这一步让它有了持续的"自我"。

Phase 1 的脚本每次运行都从零开始。上一次对话说了什么、做了什么，全部丢失。这对一个"助手"来说是不可接受的。

第一性原理
LLM 的上下文窗口是有限的（比如 128K tokens）。但用户的对话历史可能有几万轮。所以记忆系统的核心问题是：如何在有限窗口里放入最有价值的历史信息。

两层记忆

Session（短期记忆） session/manager.py

  • 每个会话的完整历史存成 JSONL 文件（追加写入，原子操作 + fsync）
  • 重放时截取最近 N 条消息（根据模型上下文窗口动态计算）
  • 当上下文过长时，自动压缩（AutoCompact）——让 LLM 自己总结之前的对话，用摘要替代原文

Memory（长期记忆） agent/memory.py

  • SOUL.md — Agent 的"人格"和行为指南
  • USER.md — 关于用户的信息（偏好、角色、习惯）
  • MEMORY.md — 跨会话的事实记录
  • Dream 整合——定期用 LLM 回顾近期对话，提取值得记住的信息写入上述文件
Session (短期) Memory (长期) ───────────────── ────────────────── 对话历史 (JSONL) SOUL.md (人格) ↓ 太长时 USER.md (用户画像) AutoCompact (LLM总结) MEMORY.md (事实) ↓ ↑ 截取最近N条 → 放入上下文 Dream (定期整合) ↑ 回顾近期对话历史

为什么要 Dream 机制？

人类在睡眠中整合白天的记忆——这就是 Dream 的类比。Agent 不能每次对话都把所有历史塞进上下文（窗口不够），但又需要记住重要的事情（用户的偏好、项目的约定）。Dream 让 LLM 自己决定什么值得长期记忆，什么可以遗忘。

AutoCompact 的精妙之处

当对话太长快撑爆上下文窗口时，让 LLM 自己总结之前的对话，然后用摘要替代原始消息。这相当于"你还记得我们之前聊了什么吗？"——LLM 说"记得，我们讨论了 X、Y、Z"，然后丢掉原始的几百条消息。记忆质量靠 LLM 的理解能力保证。

Phase 5

消息总线与渠道

Agent 不应该知道用户从哪来——Telegram、Discord 还是网页，都是"一条消息"。

到 Phase 4 为止，你有了一个能记住上下文、调用工具的 Agent。但它只能通过命令行交互。用户想从 Telegram 给它发消息怎么办？

设计原则
用一个异步消息队列（MessageBus）把"消息从哪来"和"消息怎么处理"完全解耦。Agent 核心只消费 InboundMessage，不关心它来自哪个平台。

MessageBus

bus/queue.py 只有 45 行代码——两个 asyncio.Queue，一个入站一个出站。这是整个系统最简单也最关键的组件。

bus/queue.py — 整个 MessageBus 就这么多
class MessageBus:
    def __init__(self):
        self.inbound: asyncio.Queue[InboundMessage] = asyncio.Queue()
        self.outbound: asyncio.Queue[OutboundMessage] = asyncio.Queue()

    async def publish_inbound(self, msg):   # Channel → Bus
        await self.inbound.put(msg)

    async def consume_inbound(self):       # Bus → AgentLoop
        return await self.inbound.get()

    async def publish_outbound(self, msg):  # AgentLoop → Bus
        await self.outbound.put(msg)

    async def consume_outbound(self):      # Bus → Channel
        return await self.outbound.get()

Channel 抽象

channels/base.py 定义了 BaseChannel：监听外部平台、把平台消息转成统一的 InboundMessage、把 OutboundMessage 转回平台格式发出去。

和 Tools 一样，Channels 也通过 pkgutil 自动发现。加一个新平台 = 写一个继承 BaseChannel 的文件。

为什么用消息队列而不是直接调用？

  • 解耦——Channel 和 AgentLoop 互不知道对方存在，通过 Bus 间接通信
  • 多渠道并发——15 个 Channel 同时运行，消息全部进同一个队列，AgentLoop 串行消费
  • 背压控制——如果 Agent 处理慢，消息在队列里排队，不会丢失也不会压垮 Agent
  • 测试友好——测试时不需要启动真实的 Telegram bot，直接往 Bus 里塞消息就行
Phase 6

上下文构建

你喂给 LLM 的 system prompt 决定了 Agent 的"人格"和能力边界。这是最容易被低估的部分。

核心认知
Agent 的行为不是由代码决定的，而是由 prompt 决定的。代码只是执行引擎，prompt 才是 Agent 的灵魂。

每次 Agent 处理一条消息时，ContextBuilder agent/context.py 组装完整的上下文：

System Prompt 组装顺序 ───────────────────────── 1. SOUL.md → Agent 的人格、行为准则 2. USER.md → 当前用户的画像 3. MEMORY.md → 长期记忆条目 4. Skills → 已加载的 Skill 描述 5. Tool 定义 → 所有可用工具的 JSON Schema 6. Runtime 上下文 → 当前时间、工作目录、平台信息 7. 对话历史 → 最近 N 轮（经过压缩/截断）

这就像你给一个新员工的入职手册：你是谁、用户是谁、你记得什么、你会什么技能、你手上有哪些工具、现在是几点。有了这些信息，LLM 才能做出恰当的判断。

为什么 SOUL.md 和 USER.md 是 Markdown 文件而不是代码？

因为非技术用户也需要能修改 Agent 的行为。改一行 Markdown 就能让 Agent "从今天起用中文回复"或"不要主动推荐其他工具"——不需要懂 Python，不需要重启服务。这也是为什么记忆系统用文件而不是数据库——人类可读、可编辑、可 git 版本控制。

Skills 系统

Skills agent/skills.py 是"可选的能力包"。不像 Tools 是给 LLM 调用的函数，Skills 是一段提示词——它告诉 LLM 在特定场景下应该怎么做。比如一个 "github" skill 会教 Agent 如何正确创建 PR。Skills 通过目录扫描自动加载，和 Tools/Channels 一样的插件机制。

Phase 7

WebUI

给 Agent 一张脸——浏览器里的聊天界面。

到这一步，Agent 已经能通过 Telegram、Discord 等渠道工作了。但你还需要一个自己的界面——不依赖第三方平台、能展示富文本和代码、能管理设置。

技术选型

Vite + React SPA

纯前端应用，通过 WebSocket 与后端通信。构建产物直接打进 Python 包。

nanobot 采用的方案
Next.js 全栈

前后端一体。但 nanobot 后端是 Python，不需要 Node.js 的 SSR 能力。

为什么选 Vite + React 而不是 Next.js？

nanobot 的后端是 Python（aiohttp），不需要 Node.js 服务端。WebUI 只是一个纯客户端，通过 WebSocket 连接到 Python Gateway。Next.js 的 SSR、API Routes 这些服务端能力完全用不上，反而多引入一个 Node.js 运行时。Vite + React 是最轻量的组合——构建出来的 dist/ 是纯静态文件，直接由 Python 提供服务。

WebSocket 多路复用

Agent 生成回复时是流式的——一个字一个字吐出来。HTTP 做不了这个（请求-响应模型），所以用 WebSocket 建立长连接：

浏览器 (React SPA) ↕ WebSocket 长连接 Gateway (Python, 端口 8765) │ ├── 消息收发（聊天内容） ├── 流式输出（Agent 边想边发） ├── 工具执行进度（正在读文件...） └── 会话管理（切换/创建/删除）

一个 WebSocket 连接承载多种消息类型——这就是"多路复用"。Gateway 在中间做路由：前端来的消息转发给 AgentLoop 的 MessageBus，AgentLoop 的输出流式推回前端。

Phase 8

生产化

从"能跑"到"能用"——配置、安全、API 兼容、子 Agent。

配置系统

config/schema.py 用 Pydantic 定义配置结构，从 ~/.nanobot/config.json 加载。支持 camelCase 别名（JSON 友好）。这是框架的"控制面板"——用哪个模型、开哪些渠道、工具权限等等。

安全

Agent 能执行 Shell 命令、读写文件——这意味着它有真实的破坏力。

  • 沙盒 agent/tools/sandbox.py ——Shell 命令在受限环境中执行
  • 工作区范围——限制文件操作在指定目录内
  • PTH 文件守卫 security/ ——防止代码注入
  • Pairing 机制——DM 消息需要验证码才能交互，防止陌生人操控你的 Agent
安全是 Agent 特有的问题

传统 Web 应用的安全是防止外部攻击（SQL 注入、XSS）。Agent 的安全问题是：你给了 LLM 真实的系统权限，而 LLM 可能被 prompt injection 操纵。所以安全边界不是"防黑客"，而是"限制 Agent 能做什么"——即使 LLM 被骗了，沙盒也能防止真正的损害。

OpenAI 兼容 API

api/server.py 提供 /v1/chat/completions 和 /v1/models 端点，兼容 OpenAI API 格式。这意味着任何支持 OpenAI API 的工具（Cursor、Continue 等）都可以直接连接你的 Agent——它在外部看来就像一个自定义的 "模型"。

子 Agent

agent/subagent.py + agent/tools/spawn.py ——Agent 可以生成子 Agent 来并行处理复杂任务。主 Agent 负责拆解和协调，子 Agent 各自独立执行。这是 Agent 从"单线程助手"进化为"团队协调者"的关键。

定时任务

agent/tools/cron.py ——让 Agent 自己安排定时任务。"每天早上 9 点检查一下那个 API 的状态"——Agent 会创建一个 cron job，到点自动执行。

全景回顾

从 50 行脚本到完整框架，每一步都是对前一步自然出现的问题的回答。没有过度设计，只有"用到才加"的演进。

演进逻辑链

Phase 0 Agent = LLM + Tools + Loop ↓ "我需要一个能跑的原型" Phase 1 50 行 while True 循环 ↓ "工具写死在 if/else 里太丑了" Phase 2 Tool 基类 + Registry + 自动发现 ↓ "不想绑死一个 LLM 提供商" Phase 3 Provider 抽象 + Factory ↓ "每次对话都忘记之前说过什么" Phase 4 Session 持久化 + Memory + Dream ↓ "想让多个平台的用户同时使用" Phase 5 MessageBus + Channel 插件 ↓ "Agent 的行为应该由 prompt 控制，不是代码" Phase 6 ContextBuilder + SOUL/USER/Skills ↓ "需要一个自己的界面" Phase 7 Vite + React + WebSocket Gateway ↓ "要安全、可配置、可扩展" Phase 8 Config + Security + API + Subagent
带走一句话
Agent 框架的本质是一个"最小循环 + 最大扩展性"的架构。循环本身永远是那个 while True——send → tool_call → execute → feed back。所有的工程复杂度都在循环外面：怎么插拔工具、怎么切换模型、怎么持久记忆、怎么连接多个平台。理解了循环，就理解了 Agent。
